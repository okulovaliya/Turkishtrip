# Squad Challenge — настройка общих данных (Firebase)

По умолчанию приложение хранит всё в `localStorage` браузера — каждый человек видит только свою активность на своём телефоне. Чтобы команда видела активность друг друга (общая лента, командный прогресс), нужно подключить бесплатную базу Firebase Realtime Database. Настройка разовая, займёт 5–10 минут, делает её один человек — например, тот, кто создаёт первую команду.

С этой версии приложение больше не привязано к трём заранее заведённым именам: любой человек может зарегистрироваться со своим email и либо создать свою команду (название + дата поездки), либо присоединиться к существующей по короткому коду-приглашению. Исходные tanya/lilu/nastya продолжают работать как раньше — их аккаунты и данные автоматически привязываются к их прежней команде при первом входе после обновления.

## Шаг 1. Создать проект Firebase

1. Откройте [console.firebase.google.com](https://console.firebase.google.com) и войдите с любым Google-аккаунтом.
2. Нажмите **Add project** / «Создать проект», придумайте название (например, `squad-challenge`), пройдите мастер (Google Analytics можно отключить — не нужен).

## Шаг 2. Включить Realtime Database

1. В левом меню откройте **Build → Realtime Database**.
2. Нажмите **Create Database**.
3. Выберите любой регион (например, ближайший к вам).
4. На шаге про правила безопасности выберите **Start in test mode** — это временно открывает чтение/запись всем, у кого есть ссылка на базу. Для узкого круга друзей этого достаточно; более строгие правила — см. раздел «Безопасность» ниже.

## Шаг 3. Получить конфигурацию веб-приложения

1. Нажмите на шестерёнку рядом с «Project Overview» → **Project settings**.
2. На вкладке **General** прокрутите до **Your apps** → нажмите значок `</>` (Web).
3. Введите любое имя приложения → **Register app**. Ничего дополнительно устанавливать не нужно.
4. Firebase покажет объект вида:

```js
const firebaseConfig = {
  apiKey: "AIza...",
  authDomain: "squad-challenge-xxxxx.firebaseapp.com",
  databaseURL: "https://squad-challenge-xxxxx-default-rtdb.firebaseio.com",
  projectId: "squad-challenge-xxxxx",
  storageBucket: "squad-challenge-xxxxx.appspot.com",
  messagingSenderId: "...",
  appId: "..."
};
```

5. Скопируйте эти значения в файл **`firebase-config.js`** (лежит рядом с `index.html`), заменив placeholder-строки. Нужен именно объект `firebaseConfig` — если Firebase выдал код с `import { initializeApp } from "firebase/app"`, эту строку и вызов `initializeApp(...)` не копируйте, оставьте только сам объект (`app.js` подключает Firebase самостоятельно).

## Шаг 4. Включить вход по паролю (Firebase Authentication)

Это то, что убирает пароли из файлов проекта — вместо этого их безопасно хранит и проверяет сам Firebase.

1. В левом меню откройте **Build → Authentication → Get started**.
2. На вкладке **Sign-in method** выберите **Email/Password** → включите переключатель **Enable** → **Save**.
3. Больше ничего создавать вручную не нужно: любой человек регистрируется сам, прямо в приложении, своим email и придуманным паролем (от 6 символов) — аккаунт автоматически создаётся в Firebase. При следующих входах используется тот же пароль.

## Шаг 5. Разослать приложение

Сохраните и раздайте всем нужным файлам одну и ту же папку (`index.html`, `style.css`, `app.js`, `users.json`, `firebase-config.js`, папку `avatars/`) — например, залив на любой бесплатный статический хостинг (GitHub Pages, Netlify, Vercel) и разослав ссылку, либо просто переслав папку файлов, если открываете локально в Firefox (см. примечание про CORS ниже).

После этого на экране «Профиль» появится плашка **☁️ синхронизировано со всеми** — значит облако подключено. Если конфиг не заполнен, будет **📱 только на этом устройстве**.

## Как работает регистрация и команды

- Первый вход: человек вводит имя, email и придуманный пароль. Приложение сначала пробует войти этим email/паролем, а если такого аккаунта ещё нет — сразу регистрирует его. Дальше — только email + тот же пароль.
- Сразу после входа приложение проверяет, есть ли у этого аккаунта команда. Если нет — показывает экран «Создать команду» (название + дата поездки) или «Есть код» (присоединиться к чужой команде по короткому коду).
- Код приглашения своей команды виден в любой момент на вкладке **Профиль** — там же кнопка «Скопировать», чтобы переслать его друзьям.
- Исходные три аккаунта (tanya/lilu/nastya) не должны ничего делать по-новому — при первом входе после этого обновления приложение само узнаёт их по email и привязывает к их прежней команде и данным, без создания/присоединения вручную.
- Один человек может состоять сразу в нескольких командах. В профиле — «Сменить команду» (список всех своих команд с переходом в любую), «+ Ещё команда» (создать новую или присоединиться по коду, не выходя из текущей) и «Выйти из команды» (убирает из ростера этой команды; активность и данные в её истории не удаляются). Если из команды выходит владелец (`role: "owner"`) — владение автоматически переходит следующему по дате вступления участнику. Если выходит последний оставшийся участник — команда полностью удаляется вместе с её кодом-приглашением (все её завершённые цели к этому моменту уже сохранены лично у каждого, кто в ней успел побывать, — см. ниже, так что ничего не теряется).
- Как только дата поездки проходит, цель автоматически закрывается: на главном экране появляется сводка по каждому участнику (шаги, минуты тренировок, очки — за всё время), а в профиле она навсегда остаётся в «Завершённых целях» — причём именно у человека, а не у команды: список не меняется при переключении между командами и переживает выход из команды, где цель была закрыта. Запустить новую цель (новое название, направление и дата — ростер и лента команды при этом сохраняются) может только владелец команды (`role: "owner"`) — остальные просто увидят новый отсчёт, как только он это сделает.
- Вкладка **Рецепты** — общая «поваренная книга» на всё приложение, а не отдельно для каждой команды: один и тот же список виден независимо от того, в какой команде человек сейчас состоит, и переживает переключение и выход из команды. Редактировать и удалять рецепт может только тот, кто его создал — чип «Мои рецепты» в журнале фильтрует именно по этому признаку.
- Если человек состоит сразу в нескольких командах, каждая добавленная им запись шагов/тренировки автоматически засчитывается во всех его командах, а не только в той, что открыта сейчас, — как будто он залогировал её в каждой отдельно. Позже отредактировать или удалить такую запись можно из любой команды — изменение применится сразу везде. На уже существующую (добавленную до этого обновления) активность это не распространяется — только на новые записи.
- Стрик, очки, шаги и награды в профиле принадлежат человеку, а не открытой сейчас команде: при переходе в новую или дополнительную команду они не обнуляются — приложение подтягивает актуальные цифры из всех остальных команд, где человек ещё состоит, плюс сохранённые итоги по тем, что уже закрыты или покинуты. Аватар, градиент и дневная цель шагов точно так же переносятся в новую команду с последней сохранённой версии, а не сбрасываются на стандартные.
- В профиле есть переключатель «Тема оформления» (🌙 Тёмная / ☀️ Светлая) — это настройка конкретного телефона/браузера, а не аккаунта, и не синхронизируется через Firebase, как и на большинстве других приложений. Выбор запоминается на устройстве и применяется ещё до первой отрисовки страницы, чтобы не было вспышки не той темы при открытии.

## Как это работает (данные)

- `users.json` хранит только запасной список участниц на случай проблем с сетью — **паролей там нет и никогда не было**. Пароли целиком под управлением Firebase Authentication: он их хеширует и хранит на своей стороне, ни приложение, ни файлы проекта их никогда не видят.
- В Firebase Realtime Database синхронизируются только сами данные челленджа: активность (шаги/тренировки), комментарии и реакции в ленте. Обновления приходят почти мгновенно на все устройства.
- Если `firebase-config.js` не настроен (стоят placeholder-значения), приложение переключается в локальный демо-режим на одном устройстве — без пароля вообще, и только для исходных трёх имён (см. предупреждение на экране входа). Как только конфиг заполнен и Email/Password включён в консоли — включается полноценная регистрация и создание команд.
- Данные каждой команды хранятся отдельно, в своём изолированном узле `teams/{id}/...` (активность, комментарии, реакции, участники, дата поездки) — команды не видят данные друг друга. Исходная команда живёт в `teams/turkey-2026/...`; при первом открытии обновлённой версии приложение один раз само переносит уже накопленные данные из старой плоской структуры туда — вручную ничего делать не нужно.
- Кто в какой команде состоит и под каким именем — отдельно, в `users/{аккаунт}/memberships/{id команды}`. Короткие коды-приглашения резолвятся в `inviteCodes/{код}`.
- Рецепты — единственные данные, которые лежат вообще вне `teams/...`: общий узел `recipes/{id}` в корне базы, один на всё приложение. Автор определяется по Firebase Auth uid (не по имени в конкретной команде), поэтому «моё» остаётся «моим» при переходе между командами.

## Безопасность базы данных

По умолчанию (Realtime Database → Rules) Firebase создаёт **test mode** правила, которые открывают чтение и запись всем, у кого есть ссылка на базу, но только временно (обычно 30 дней от создания базы). После того как вы включили Firebase Authentication (Шаг 4), обязательно замените их — теперь, когда в приложении может быть несколько независимых команд, это уже не «на будущее», а нужно сразу.

Минимальный вариант (как раньше) — пускать любого, кто вошёл в приложение, к любым данным:

```json
{
  "rules": {
    ".read": "auth != null",
    ".write": "auth != null"
  }
}
```

У него есть реальный минус: с этим правилом человек из одной команды технически может прочитать или изменить данные **чужой** команды, если каким-то образом узнает её `teamId` (например, из ссылки-приглашения). Для одной команды на всех (как было раньше) это не проблема, но теперь, когда команд может быть несколько, лучше использовать более строгий вариант — он проверяет, что человек действительно состоит в той команде, к данным которой обращается:

```json
{
  "rules": {
    "teams": {
      "$teamId": {
        ".read": "auth != null && root.child('users/' + auth.uid + '/memberships/' + $teamId).exists()",
        ".write": "auth != null && root.child('users/' + auth.uid + '/memberships/' + $teamId).exists()"
      }
    },
    "users": {
      "$uid": {
        ".read": "auth != null && auth.uid === $uid",
        ".write": "auth != null && auth.uid === $uid"
      }
    },
    "inviteCodes": {
      ".read": "auth != null",
      "$code": {
        ".write": "auth != null && (!data.exists() || newData.val() === null)"
      }
    },
    "recipes": {
      ".read": "auth != null",
      "$recipeId": {
        ".write": "auth != null && (!data.exists() || data.child('authorUid').val() === auth.uid)"
      }
    }
  }
}
```

Как это работает: `users/{uid}/memberships/{teamId}` существует ровно тогда, когда этот человек состоит в этой команде — правило просто это проверяет. `inviteCodes` читать может любой вошедший (иначе нельзя будет проверить код при присоединении); перезаписать чужой существующий код на другое значение нельзя (защита от угона ссылки-приглашения), но создать новый код или удалить свой (когда команда распускается — см. ниже) можно. `recipes` читать может любой вошедший (общая книга рецептов); создать новый рецепт может кто угодно, а вот изменить или удалить уже существующий — только тот, чей `authorUid` в нём записан. Обновите правила в Firebase Console → Realtime Database → вкладка **Rules** → вставьте нужный JSON → **Publish**.

При выходе из команды (`leaveTeam` в `app.js`) запись в `teams/{teamId}/members/{login}` и в `users/{uid}/memberships/{teamId}` намеренно удаляются двумя отдельными вызовами, а не одним `update()`: при groupped-обновлении правило для `teams/{teamId}` проверяется уже по состоянию *после* всего обновления целиком, и к этому моменту `memberships/{teamId}` уже был бы удалён тем же вызовом — проверка `.exists()` не прошла бы, и Firebase отклонил бы собственный выход человека из команды.

## Открытие без сервера

При открытии `index.html` напрямую двойным кликом (`file://`) браузер может блокировать `fetch('users.json')` из-за CORS — в этом случае сработает встроенный резервный список участниц в `app.js` (имена и цвета, без паролей), сам вход по-прежнему пойдёт через Firebase Authentication. Firebase-скрипты грузятся с CDN и не подвержены этому ограничению. Рекомендуется всё же открывать через любой локальный сервер (`python3 -m http.server`) или хостинг — так работает стабильнее.
